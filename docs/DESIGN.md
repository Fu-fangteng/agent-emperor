# 设计说明

> 这套框架"为什么这么设计"。操作指引看 ../README.md。

## 一句话

把"一个人编排、多个 AI agent 接力开发"的流程，从写死在 CC+Codex 里，
抽象成**工具无关的协作契约 + 每个工具的薄适配层**，使其可 clone 复用、可自定义编制、可接入新工具。

## 四个关键设计决定

1. **角色与工具解耦**：role（要干的活）配置在 `team.yaml`，tool（谁来干）通过 adapters 适配。换工具不改流程。
2. **生成式适配（A 方案）**：适配文件由 team.yaml 生成静态文件，任何工具原生可读，不依赖运行时解析 yaml。改配置即重新生成，生成本身是一道确认节点。
3. **人工编排是 feature**：每个 STATUS 节点 = 审查点 + commit 点 + 交接点。交接已退化成"复制一段写好的 prompt"，所以可控性几乎免费。不追求全自动流转。
4. **文件即真相 + 一 role 一对话**：跨 agent 一切落文件；每个 role 用独立对话保上下文干净、保独立视角（避免 agent 审自己写的代码）。

## 继承关系（父类 / 子类）

本仓是**最原始的父类**——干净、通用、不含任何具体项目的私货。
真实项目（如 atoms_benchmark）是**子类**：clone 本仓后填 team.yaml、加项目特有的 global 规则，本地改造继承而来。
父类的更新可以重新下发给子类；子类的项目私货不回流污染父类。

## 血缘

源自内部多 Agent 协作实践（Codex 开发 + Claude Code 审查），及《Goal-Loop-AI 长期协作工作法》。
设计对标业界两条主线：Spec-Driven Development（规格即唯一事实源）+ 文件黑板/Handoff 协作模式。
