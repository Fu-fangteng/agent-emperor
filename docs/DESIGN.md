# 设计说明

> 这套框架"为什么这么设计"。操作指引看 ../README.md。

## 一句话

把"一个人编排、多个 AI agent 接力开发"的流程，从写死在 CC+Codex 里，
抽象成**工具无关的协作契约 + 数据驱动的角色编制 + 双重硬约束的运行期保护**，使其可 clone 复用、可自定义编制、可接入新工具。

## 六个关键设计决定

1. **角色与工具解耦**：role（要干的活）配置在 `team.yaml`，tool（谁来干）通过 `core/generate.py` 渲染成各工具原生吃的适配文件（CC 的 CLAUDE.md、Codex 的 AGENTS.md）。换工具不改流程。
2. **生成式适配（A 方案）**：适配文件由 team.yaml 生成静态文件，任何工具原生可读，不依赖运行时解析 yaml。改配置即重新生成，生成本身是一道确认节点。
3. **双重硬约束**：每个 worker 接到 prompt 第一步跑两道锁——时序锁（当前 STATUS 轮到我吗）+ 能力锁（这事在我职责内吗、能不能改源码，由 `can_write_code` 字段决定）。任一不过立刻停手，挡住"在错误的窗口里下错指令"。
4. **角色数据驱动、零硬编码**：通用 `/act` 触发器从 team.yaml 现读角色，绝不幻觉出未定义的角色。plan/dev/review 只是示例编制，可整套替换为安全审计、内容运营等任意角色组合。
5. **人工编排是 feature**：每个 STATUS 节点 = 审查点 + commit 点 + 交接点。交接已退化成"复制一段写好的 prompt"，可控性几乎免费。不追求全自动流转。
6. **文件即真相 + 一 role 一对话**：跨 agent 一切落文件；每个 role 用独立对话保上下文干净、保独立视角（避免 agent 审自己写的代码）。

## 继承关系（父类 / 子类）

本仓是**最原始的父类**——干净、通用、不含任何具体项目的私货。
真实项目是**子类**：clone 本仓后填 team.yaml、加项目特有的规则。

**三层模型**保证父类迭代不污染子类业务成果：
- **框架层**（`core/framework-manifest.txt` 列出）：`--upgrade` 时镜像覆盖、删废弃。
- **实例层**：`team.yaml`、业务代码、phases 产物、本机配置——绝不触碰。
- **生成层**：`CLAUDE.md`、`AGENTS.md`——升级后跑一次 `core/generate.py` 重新生成。

## 血缘

源自内部多 Agent 协作实践（Codex 开发 + Claude Code 审查），及《Goal-Loop-AI 长期协作工作法》。
设计对标业界两条主线：Spec-Driven Development（规格即唯一事实源）+ 文件黑板/Handoff 协作模式。
