Agent 皇帝（Agent Emperor）

多 Agent 协作框架 v2

皇帝不是亲自搬砖的人，而是低频编排器：所有想法先交给前台分拣，再由 worker 窗口按角色接力。你只做三件事：看一眼、复制 prompt、开新对话粘贴；上下文、流程记忆和范围约束都落在文件总线上。

功能亮点

把多 Agent 协作包装成一套可审查的产品体验
01 / STATUS 接力

每一步只看 handoff 顶部状态，谁该干活、谁该审查，一眼就能知道。

皇帝只做低频决策

看一眼前台或 worker 给出的交接内容，复制虚线之间的 prompt，开新对话粘给下一位角色。

前台不下场

/setup-team 只做配置、澄清和路由，产出转达 prompt 后停住，不自动驱动下一棒。

每次交接都有证据

requirements、plan、handoff、review 都在仓库里，审查者不用翻聊天记录。

协作范式可伸缩

从两角色精简接力到四角色完整流水线，AE 用同一套 STATUS、文件总线和转达 prompt 保持秩序。

三个核心概念

Front Desk / Worker / 转达 prompt

前台 Front Desk

人唯一打交道的入口。第一次用来配团队；之后接收想法、澄清需求、判断该交给哪个角色。

/setup-team
配置团队 / 分拣请求 / 产出 prompt
不写代码、不写方案、不写 review

Worker

真正干活的 agent 窗口。每个角色独立对话，先读共享文件和 STATUS，再按职责修改或审查。

/act
读 team.yaml + handoff.md
过时序锁 + 能力锁
靠文件接力，不靠聊天记忆

转达 prompt

前台和 worker、worker 和 worker 之间唯一传递物。格式固定，方便人复制到下一扇窗口。

✅ 我（项目 · role）的活干完了...
👉 请把下面这段复制给 next_role...
—————（复制从这里开始）—————

配置阶段 → 开发循环 → 收尾归档

① 配置阶段

拿到框架后运行 ./init.sh，再对前台敲 /setup-team。前台问项目名、角色、工具、角色归属、交接逻辑和身份锚点，生成 team.yaml 与工具适配文件。

./init.sh
/setup-team
生成 team.yaml + CLAUDE.md / AGENTS.md
关掉旧 worker 窗口后再开工

② 开发循环

把想法交给前台，前台分拣出转达 prompt；人复制给对应 worker。worker 过两道锁后干活，更新 handoff，再产出下一棒 prompt。

我有个想法
→ 前台产出 prompt
→ 新开 worker 对话
→ worker 干活并交下一棒

③ 收尾归档

当状态机走到 next_role: null，本增量收尾。此时是天然提交点，可以 commit 并开启下一轮增量。

STATUS: APPROVED
轮到: （收尾 / 可合并归档）
git add . && git commit

核心机制

        安全锁、文件真相、身份探测器
      

      

        

          
官职与工具解耦

          
team.yaml 是唯一事实源：agent 名字、工具、role、能力锁和 handoff 都从这里生成。

          
roles:
  - name: developer
    can_write_code: true
  - name: reviewer
    can_write_code: false
agents:
  - name: cc
    roles: [reviewer]
  - name: codex
    roles: [developer]

        

        

          
文件总线即奏折

          
需求、方案、实现交接和审查都写进总线文件；聊天只负责把下一卷奏折交出去。

          
docs/agent-collaboration/
  phases/v2/
    requirements.md
    plan.md
    handoff.md
    review.md

        

        

          
双重硬约束

          
每轮开场先核对时序锁和能力锁；没轮到就停，role 不具备写代码权限也停。

          
时序锁: STATUS 轮到我
能力锁: can_write_code 允许当前任务

        

        

          
身份玉玺

          
每轮结尾固定署名，丢了 role 或产品名就能立刻发现上下文漂移。

          
🎩 您的「multiagent-workflow-demo · developer」已为您效劳完毕，主人。

        

      

    


    

      

        
触发器速查

        配置 → 干活 → 自检 → 对齐
      

      

        

          
/setup-team

          
你对前台敲。没配过就登记团队并生成配置；配过了就分拣请求，产出转达 prompt。

          
谁敲: 你（对前台）
干嘛: 配置 / 分拣 / 产出 prompt

        

        

          
/act

          
worker 窗口敲。按 team.yaml 和当前 STATUS，以轮到的角色身份干活，先过两道锁。

          
谁敲: worker 窗口
干嘛: 现读角色 / 职责 / 读写范围

        

        

          
/whoami

          
任意窗口敲。当场自检“我担任什么角色、轮没轮到我、当前 STATUS、下一步”。

          
谁敲: 任意窗口
干嘛: 自检身份
性质: 纯只读

        

        

          
/sync

          
任意窗口敲。看全局卡在哪个 STATUS、轮到谁、你该切到哪个 agent。

          
谁敲: 任意窗口
干嘛: 全局对齐
性质: 纯只读

        

      

    


    

      

        
进阶能力

        升级父类框架，多产品并行，自定义状态机
      

      

        

          
框架升级

          
老项目同步 Agent Emperor 新能力时，跑 ./init.sh --upgrade /path/to/project。框架层镜像覆盖，实例层绝不触碰，生成层重生。

          
框架层: core/ + skills 按 manifest 覆盖
实例层: team.yaml / 业务代码 / phases 不动
生成层: CLAUDE.md / AGENTS.md 重新生成

        

        

          
同时跑多个产品

          
推荐每个产品独立子目录，各跑一次 init.sh 和 /setup-team，总线物理隔离。共享 codebase 时，用不同 bus.dir 分开。

          
推荐: product-a/ + product-b/
共享: docs/agent-collab-p1
      docs/agent-collab-p2

        

        

          
自定义角色

          
planner / developer / reviewer 是默认编制。安全审计可以换成 analyst / pentester / auditor，内容运营可以换成 editor / writer / fact-checker。

          
roles:
  - name
  - desc
  - can_write_code

        

        

          
自定义状态机

          
handoff 定义每个 STATUS 交给哪个角色；next_role: null 表示流程收尾。

          
handoff:
  - state: DEV_DONE
    next_role: reviewer
  - state: APPROVED
    next_role: null
